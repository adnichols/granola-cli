Granola CLI Auth Recovery PR 1 Plan

Make granola auth login and subsequent API commands behave truthfully for supported plaintext desktop auth: prefer stored-accounts.json, preserve legacy supabase.json, detect stale plaintext beside newer encrypted state, stop false-success messaging, and report refresh/auth failures with actionable diagnostics.

Repo: granola-cli Source: docs/AUTH-RECOVERY-PLAN.md Durable encrypted/companion auth: deferred research

Status

Execution-ready for the first shippable PR. This plan intentionally scopes implementation to plaintext credential import, stale/encrypted-state diagnostics, refresh/error truthfulness, and documentation. It does not implement encrypted-state decryption or Granola companion socket support.

Execution boundary: P1–P4 below are the only active implementation phases. Durable modern auth must be handled by a later reviewed research/decision plan because it changes the package trust boundary.

Goal

Stop the current false-success loop where auth login imports stale plaintext credentials, prints an unqualified success message, and meeting list immediately fails with Authentication required.

The PR 1 result should either import usable plaintext credentials and say exactly which source was used, or clearly label the result as import-only / unsupported encrypted state so users and agents do not retry the same ineffective command.

Why this plan exists

Modern Granola desktop installs can leave legacy plaintext files behind while updating encrypted files. The current CLI reads only supabase.json on macOS, treats parsing as authentication success, and on API rejection tells the user to run the same login command again.

PR 1 fixes the misleading behavior without crossing the encrypted-state trust boundary.

Current implementation reality

Auth library

src/lib/auth.ts stores credentials in keychain service com.granola.cli, account credentials. It parses WorkOS/Cognito/root-token variants from supabase.json only.

Login command

src/commands/auth/login.ts calls loadCredentialsFromFile(), saves any parsed credentials, and prints success without source metadata, stale-state warnings, or server validation.

Refresh path

refreshAccessToken() posts directly to https://api.workos.com/user_management/authenticate. Failure returns null; the caller rethrows the original 401 and global error handling recommends login again.

Tests

Existing tests cover supabase parsing, keychain storage, login command success/failure, and token refresh success/failure. They do not cover stored-accounts.json, stale plaintext, encrypted-only diagnostics, or false-success wording.

Product intent alignment

  • Golden path: granola auth login selects the freshest supported plaintext source automatically, preferring stored-accounts.json over legacy supabase.json.
  • Truthful messaging: importing credentials is not the same as proving API usability. If encrypted siblings are newer and no validation succeeds, login output must be explicitly import-only or warning-qualified.
  • Self-healing: when credentials can be safely refreshed or re-imported from supported plaintext, the CLI should do that and leave keychain state truthful.
  • Fail closed: when only encrypted desktop state appears current, the CLI must not guess, decrypt, or log secrets. It should explain unsupported encrypted state with file names and next action.
  • Agent-legible errors: errors must state what failed, what the CLI tried, why retrying login may not help, and what to do next.

Locked decisions

  • stored-accounts.json is preferred over legacy supabase.json when both parse successfully.
  • auth login success must identify the imported source path/type.
  • auth login must not imply API usability unless validation succeeds. If plaintext is stale relative to encrypted state and validation is not performed or fails, output must clearly label the result as import-only / may be stale instead of an unqualified Credentials imported successfully.
  • auth status remains credential-presence-only unless a separate validation feature is explicitly scoped; docs must say this.
  • Diagnostics may include path basenames, existence, mtimes, and relative freshness; never file contents or token-like values.
  • If the current refresh path rejects tokens, the CLI should emit a specific refresh-failed/stale-credentials diagnostic rather than blindly recommending the same login command.

Acceptance criteria

  1. AC1: On installs with current plaintext stored-accounts.json, granola auth login imports from stored-accounts.json before falling back to supabase.json, and a subsequent lightweight API command succeeds when the imported credentials are valid.
  2. AC2: Legacy installs with only usable supabase.json continue to import and authenticate.
  3. AC3: When plaintext credentials import but are older than encrypted siblings, login must not present unqualified Credentials imported successfully as a complete fix unless validation succeeds; otherwise the output must clearly label the action as import-only / possibly stale.
  4. AC4: When encrypted state exists but supported plaintext does not parse, login reports encrypted-only desktop state as unsupported and names relevant files without secrets.
  5. AC5: On 401 after refresh failure, the CLI distinguishes stale/rejected credentials from missing credentials and does not create a login retry loop.
  6. AC6: Documentation explains source preference, stale/encrypted diagnostics, refresh failure behavior, and that auth status is credential-presence-only unless validation is later added.
  7. AC7: Tests cover stored accounts parsing, legacy fallback, stale plaintext import-only messaging, encrypted-only diagnostics, refresh failure messaging, and secret-safe diagnostic boundaries.

BDD scenarios

S1 — Fresh stored-accounts plaintext

Given stored-accounts.json contains fresh usable tokens and supabase.json is stale or absent, when the user runs granola auth login, then the CLI imports from stored-accounts.json, stores credentials in keychain, reports the source, and granola meeting list --limit 1 succeeds in manual smoke testing.

S2 — Legacy supabase plaintext

Given only supabase.json exists and contains usable legacy tokens, when login runs, then the CLI imports those credentials, reports the legacy source, and preserves existing legacy behavior.

S3 — Stale plaintext with newer encrypted state

Given stored-accounts.json or supabase.json parses but stored-accounts.json.enc, supabase.json.enc, or cache-v6.json.enc is newer, when login runs without successful validation, then the CLI may save the imported credentials but must label the result as import-only / possibly stale and must not print an unqualified success message.

S4 — Encrypted-only state

Given no supported plaintext credentials parse and encrypted desktop files exist, when login runs, then it fails closed with unsupported encrypted-state diagnostics and no secret output.

S5 — Refresh endpoint rejects token

Given the access token is expired and refresh returns 400 or 401, when an API command retries, then the CLI reports refresh rejection/stale credentials and explains what was tried instead of only saying to run granola auth login.

S6 — No desktop state

Given none of the supported plaintext or encrypted Granola desktop state files exist, when login runs, then it reports that no desktop auth state was found and asks the user to install/sign into Granola desktop.

node dist/main.js auth login
node dist/main.js meeting list --limit 1 --output json --no-pager

  • Stale/encrypted-only case: login and/or the API command reports stale plaintext or unsupported encrypted-only state with file names/relative freshness and no secrets. This diagnostic is a successful PR 1 outcome on machines where current usable plaintext credentials are unavailable.

Do not paste command output containing tokens or decrypted payloads. If debug logs are needed, redact bearer tokens, access tokens, refresh tokens, DEKs, session IDs, and decrypted JSON.
